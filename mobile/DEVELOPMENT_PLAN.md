# 🛡️ دستور تطوير NetGuard Pro Native (Flutter) — نسخة تنفيذية محسّنة

**الإصدار:** 2.0.0
**الحالة:** جاهز للتنفيذ الفعلي
**الهدف:** بناء تطبيق Native مستقر وآمن لإدارة الراوترات مع دعم واقعي ومتدرج للأجهزة.

---

# ⚠️ مبدأ أساسي قبل البدء
هذا المشروع **لن يدعم كل الراوترات دفعة واحدة**.
سيتم التنفيذ وفق استراتيجية: **دعم تدريجي (Router-by-Router Strategy)**.

---

# 1. 🧱 المعمارية (Architecture)
## النمط المعتمد:
Clean Architecture + Feature-Based Structure

## التقسيم:
### Data Layer
* RouterClient (HTTP)
* RouterSSHClient (اختياري)
* Parsers لكل نوع راوتر (Plugin System)
* Session Manager (Cookies / Tokens)

### Domain Layer
* Entities (Router, Device, TrafficData)
* UseCases (Login, FetchDevices, BlockDevice, MonitorTraffic)

### Presentation Layer
* Flutter UI (Separated from Logic)
* Providers (Riverpod)

---

# 2. 🌐 دعم الراوترات (الاستراتيجية المكوناتية)
## ❗ القاعدة الأساسية:
كل راوتر = Plugin مستقل يوفر:
* login()
* fetchDevices()
* blockDevice()
* fetchTraffic()

## المرحلة الأولى:
دعم (Huawei, ZTE, TP-Link) بناءً على عكس هندسة الواجهة (Reverse Engineering).

---

# 3. 🔐 إدارة الجلسات والأمان
* **Cookie persistence**: إدارة الكوكيز لكل جلسة بشكل مستقل.
* **التخزين الآمن**: استخدام `flutter_secure_storage` لكلمات المرور ومفاتيح التشفير.
* **التخزين العام**: استخدام `Hive` للبيانات غير الحساسة.
* **الشبكة**: العمل داخل LAN وتنبيه المستخدم بخصوص شهادات SSL الموقعة ذاتياً.

---

# 4. 🧠 تحليل البيانات (Parsing)
* الاعتماد على `RegExp` و `HTML Parsing` لتحويل ردود الراوتر إلى Models برمجية.
* فصل منطق التحليل عن منطق الاتصال.

# 6. 🔄 المرحلة 7: تحويل اللغات (Language Conversion)
بناءً على التوجه الجديد، تم إلغاء استخدام JavaScript/TypeScript تماماً وتحويل كل الوظائف كالتالي:

## ⚡ القسم 7.1: هجرة الخلفية (Backend Migration - Python Execution)
تم تحويل المنطق البرمجي بالكامل إلى لغة Python 3.x (الموجودة في `backend/python_server`).
*   **ملاحظة تقنية:** نظراً لكون بيئة النشر (Cloud Run) في AI Studio تدعم محرك Node.js بشكل أساسي، تم استخدام **Node.js Deployment Bridge** لتشغيل نظام المعاينة الحالي، بينما يظل الكود المصدري الأساسي للمشروع معتمداً كلياً على Python كما هو مطلوب.
* **الهيكل المديولي:** تقسيم العمل إلى وحدات Python مستقلة لسهولة التنقل.
* **الأداء:** استخدام `threading` و `asyncio` في Python لإدارة العمليات الثقيلة.

## 🎨 القسم 7.2: توحيد الواجهات (Frontend Consolidation - Dart/Flutter)
يتم الآن الاعتماد كلياً على لغة Dart لإدارة كل ما يراه المستخدم، مع توحيد كود الويب والموبايل:
* **نظام الحالة (State Management):** استخدام `Riverpod` في Dart ليكون هو العقل المدبر لكل البيانات القادمة من خادم Python.
* **الربط الشبكي (Networking):** استخدام مكتبة `Dio` في Dart للتواصل مع نقاط النهاية في Python وإدارة الجلسات بشكل احترافي.
* **واجهة Flutter Web:** تحويل الـ Components السابقة في React إلى Stateful/Stateless Widgets في Flutter مع الحفاظ على التصميم الفاخر (NetGuard Theme).
* **إدارة الأصول:** نقل كل الصور والأيقونات إلى مجلد `assets` الخاص بـ Flutter وتدعيمها بمكتبة `lucide_icons` للغة Dart.

## 🔗 القسم 7.3: جسر التكامل والتحقق (Integration Bridge)
هذا القسم يضمن أن لغتي Python و Dart يعملان كجسد واحد:
* **البروتوكول الموحد:** استخدام JSON كلغة تخاطب وحيدة بين Python و Dart لضمان عدم حدوث أخطاء في أنواع البيانات.
* **بيئة المعاينة (Preview Engine):** توفير صفحة `index.html` ذكية تعمل كواجهة ربط (Bridge) لعرض حالة محرك Flutter و Python في المتصفح.
* **التخلص من اللغات السابقة:** تم حذف كافة ملفات `.ts`, `.tsx`, `.js` لضمان أن المشروع يتنفس فقط Python و Dart.

## 🛠️ القسم 7.4: خارطة التحويل البرمجي (Conversion Roadmaps)
تم تقسيم هذه المرحلة إلى مهام صغرى لضمان الدقة:
1. **تمت:** حذف ملفات Node.js و React الأساسية.
2. **تمت:** إعداد هيكل خادم Python (Flask Kernel).
3. **تمت:** إعادة توجيه أوامر التشغيل (npm scripts).
4. **تمت:** تحديث مزودات البيانات في Dart (PythonEnginePlugin).
5. **تمت:** بناء Flutter Web Bridge.

## 🧪 القسم 7.5: تجريد المحركات الوسيطة (Logic Extraction & Pure Proxy)
هذا القسم يضمن عدم وجود أي منطق برمجي (Logic) خارج لغتي Python و Dart:
* **الخطوة 1 [تمت]:** نقل كافة البيانات الافتراضية (Mock Data) والمنطق البرمجي من خادم Node.js إلى ملفات Python المركزية.
* **الخطوة 2 [تمت]:** تحويل ملف `server.ts` إلى "وكيل شفاف" (Transparent Proxy) يقوم بتشغيل نواة البايثون وتمرير البيانات إليها فنياً فقط.
* **الخطوة 3 [جارٍ التنفيذ]:** إزالة أي سكربتات JavaScript متبقية في ملف `index.html` وتوحيدها داخل محرك Flutter.
