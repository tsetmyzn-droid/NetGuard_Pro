# 🧠 AI Internal Execution Reference — NetGuard Pro

هذا الملف يمثل رؤيتي العملية كمهندس برمجيات (الذكاء الاصطناعي) لتحويل المتطلبات إلى نظام واقعي ومستقر.

---

## 🛠️ المعمارية التنفيذية (Execution Architecture)

سأعتمد على هيكلية "الخدمة المركزية والوحدات التابعة":

1.  **Engine Layer (`NetGuardEngine`):**
    - يعمل كمراقب (Observer) لحالة الـ Plugin النشط.
    - يقوم بعملية الـ Polling كل 2 ثانية (قابل للضبط لاحقاً).
    - يحتوي على منطق **الـ Smoothing**: (السرعة الحالية = 0.7 × السرعة الجديدة + 0.3 × السرعة السابقة) لتقليل تذبذب الجراف.
    - معالجة الـ Reset: إذا كانت القيمة الجديدة أقل من السابقة، نعتبر أن العداد بدأ من الصفر ونجمع الفرق من 0.

2.  **Plugin Layer (`RouterPlugin` System):**
    - كل Plugin سيكون عبارة عن Class مستقل يرث من `RouterPlugin`.
    - **OpenWrt:** يستخدم LuCI RPC (JSON-RPC 2.0). التحدي هنا هو الحفاظ على الـ Token في الـ Cookies وعمل Auto-Refresh.
    - **Huawei:** سيتم استخدام منطق الـ SessionId والـ Token (CSRF) المعتاد في راوترات WE، مع معالجة الـ XML/JSON حسب الموديل.

3.  **Network Layer (`SecureDioClient`):**
    - إعداد `Dio` لتجاهل أخطاء شهادات SSL (`onHttpClientCreate`).
    - إضافة Interceptor عالمي لمراقبة كود الخطأ 401 (Unauthorized) لبدء عملية الـ Auto Re-login فوراً.

## 🚀 خطة التنفيذ المتسلسلة (Step-by-Step)

### المرحلة 1: أساسيات المحرك (The Engine)
- [ ] بناء `NetGuardEngine` مع منطق حساب الـ Delta والـ Smoothing.
- [ ] تنفيذ الـ `InterfaceStatus` كنموذج بيانات موحد لجميع الـ Plugins.

### المرحلة 2: تفعيل OpenWrt (Primary Focus)
- [ ] تحويل `OpenWrtClient` ليدعم الـ Persistent Session.
- [ ] ربط الـ Plugin بالمحرك وعرض البيانات على الجراف الحقيقي.

### المرحلة 3: التوسع للراوترات المحلية (Huawei/ZTE)
- [ ] بناء Plugin لراوترات Huawei (موديلات WE) مع التركيز على الـ API الخاص بـ `device_signal` و `traffic_statistics`.

### المرحلة 4: الاستقرار والتعامل مع الحالات الحرجة
- [ ] برمجة الـ `Exponential Backoff` في حالة فشل الـ Request.
- [ ] إضافة Notification UI بسيط داخل التطبيق عند فقدان الاتصال.

## ⚠️ محاذير برمجية (Anti-Patterns to Avoid)

- **تجنب الـ Tight Coupling:** لا يجب أن يعرف الـ UI نوع الراوتر الذي يتصل به؛ يجب أن يتعامل فقط مع `NetGuardEngine`.
- **تجنب الـ UI Blocking:** كل عمليات الـ Parsing (خاصة الـ XML في Huawei) يجب أن تتم في خلفية الطلب باستخدام `Isolates` إذا كانت البيانات ضخمة، لتفشل أي محاولة لتعليق الواجهة.
- **تجنب هدر البطارية:** إيقاف الـ Polling فور خروج المستخدم من صفحة الداشبورد.

## 🎯 معايير الجودة للتسليم

1.  **الدقة:** تطابق السرعة المعروضة مع واقع الاستهلاك (بنسبة خطأ < 5%).
2.  **المرونة:** سهولة إضافة Plugin جديد في أقل من 30 دقيقة.
3.  **الاستجابة:** انتقال "سلس" للجراف بدون قفزات عشوائية (بفضل الـ Smoothing).
