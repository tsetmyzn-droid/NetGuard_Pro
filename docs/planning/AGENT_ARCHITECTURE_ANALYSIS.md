# 🔍 AGENT_ARCHITECTURE_ANALYSIS.md

بصفتي Senior Distributed Systems Architect، قمت بتحليل بنية مشروع NetGuard Pro الحالية لتقييم مدى جاهزيتها لاستيعاب نظام الـ Agent المتقدم.

## 📊 تحليل البنية الحالية (Current State)

### 1. نظام الإضافات (Plugin System)
يعتمد النظام على واجهة `RouterPlugin` (Abstract Class) التي تحدد العمليات الأساسية. هذا التصميم ممتاز لأنه يسمح بعزل منطق كل راوتر بشكل مستقل.
- **تفاعل RouterFactory**: يقوم الـ Factory بإنشاء نسخة من الإضافة بناءً على اكتشاف النوع (Discovery) أو الاختيار اليدوي.

### 2. تكامل OpenWrt الحالي
يعمل حالياً عبر بروتوكول `LuCI RPC` التقليدي (`/cgi-bin/luci/rpc/`).
- **نقاط الضعف**: الاعتماد الكلي على LuCI RPC يعني أننا مقيدون بما توفره واجهات LuCI الافتراضية. على سبيل المثال، لا يوجد حالياً "تتبع استهلاك لكل جهاز" بشكل افتراضي وسهل عبر RPC دون استعلامات معقدة ومكلفة للموارد.

### 3. إدارة التخزين (PersistenceManager)
يستخدم نظام تشفير AES متطور مع IV عشوائي و Checksum للتحقق من النزاهة.
- **الحالة الحالية**: يتم تخزين البيانات في ملف JSON مشفر (`engine_state.enc`).
- **العائق**: هذا النظام مثالي للإعدادات البسيطة، ولكنه غير فعال لتخزين "بيانات تاريخية ضخمة" (Historical Analytics) لكل جهاز على مدار أشهر. سنحتاج للانتقال إلى SQLite أو نظام تخزين تسلسلي.

### 4. محرك مراقبة المرور (NetGuardEngine)
يعتمد على آلية `Polling` (الاستعلام الدوري) التكيفية (كل 1-5 ثوانٍ).
- **الآلية**: يحسب الفرق (Delta) بين القراءات المتتالية للبايتات لتقدير السرعة اللحظية.
- **نقطة الضعف**: يتم الحساب بالكامل في جانب التطبيق (Client-side). هذا يزيد من استهلاك البطارية والبيانات. الـ Agent سيسمح بنقل جزء من هذه المعالجة للراوتر.

---

## ⚠️ المخاطر والقيود المرصودة

1. **الاقتران (Coupling)**: حالياً، منطق التواصل مرتبط ببروتوكولات HTTP الخاصة بالراوتر مباشرة. يجب إضافة طبقة "Agent Client" منفصلة.
2. **انتشار الفشل (Failure Propagation)**: إذا تعطلت خدمة RPC على الراوتر، يتوقف المحرك بالكامل. نظام الـ Agent يحتاج لآلية "Fallback" للعودة للنظام القديم تلقائياً.
3. **حدود الأمان (Security Boundaries)**: الاعتماد الحالي على الـ Cookies والـ Session IDs الخاص بـ LuCI. الـ Agent سيتطلب نظام HMAC Request Signing خاص بنا لضمان عدم التلاعب بالطلبات.

---

## 🛠️ فرص التوسع (Extension Points)

- **RouterPlugin**: يمكن إضافة ميزة `hasAgent => true` لتمكين واجهات الـ Agent.
- **Engine Data Model**: توسيع `NetGuardSystemState` ليشمل `deviceStatsHistory`.
- **Infrastructure**: البنية الحالية تدعم إضافة "Client" جديد داخل `OpenWrtPlugin` يتحدث مع الـ Agent بجانب LuCI RPC.

---
**الخلاصة**: البنية الحالية نظيفة وموديولية، مما يجعل دمج "NetGuard Agent System" عملية تطوير طبيعية وليست إعادة بناء جذرية. العقبة الأكبر ستكون في إدارة البيانات التاريخية والأمان بين العميل والراوتر.
