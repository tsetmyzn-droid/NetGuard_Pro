# 🏗️ NETGUARD_AGENT_SYSTEM_DESIGN.md

تصميم نظام **NetGuard Agent** كنظام موزَّع يهدف لتحويل الراوتر من مجرد بوابة مرور إلى مركز بيانات ذكي.

## 🧱 المكونات الأساسية (System Components)

### 1. Router-side Agent (الوكيل)
- **الهدف**: تشغيل خدمات التجميع والمعالجة محلياً على الراوتر.
- **المهام**: تجميع الإحصائيات من النواة (Kernel)، إدارة قاعدة البيانات المحلية المصغرة، والرد على طلبات التطبيق.

### 2. Local Secure API
- **الهدف**: توفير واجهة تواصل آمنة بين التطبيق والوكيل.
- **التقنية**: بروتوكول HTTP مغلف بطبقة **HMAC Signature** لضمان هوية التطبيق ونزاهة البيانات.

### 3. Telemetry Collector
- **الهدف**: تجميع البيانات الخام من `nlbwmon` و `iptables`.
- **الآلية**: يعمل في الخلفية كخدمة خفيفة تجمع البيانات كل N دقيقة وتخزنها في SQLite مؤقت.

### 4. Historical Usage Engine
- **الهدف**: تحليل البيانات التاريخية.
- **الموقع**: مشترك بين الراوتر والتطبيق (الراوتر يخزن الخام، التطبيق يحلل ويعرض).

### 5. Watchdog / Self-Recovery Layer
- **الهدف**: مراقبة صحة الوكيل.
- **الآلية**: إذا تجاوز استهلاك الذاكرة 10% من إجمالي رام الراوتر، يقوم الوكيل بإيقاف نفسه فوراً وكتابة `disabled.flag`.

---

## 🔄 تدفق البيانات (Communication Flow)

1. **الطلب (Request)**: يرسل تطبيق Flutter طلباً يحتوي على `Timestamp` و `Nonce` و `HMAC Signature`.
2. **التحقق (Verification)**: يتحقق الوكيل من التوقيع باستخدام المفتاح المشترك (Pre-shared Key).
3. **التجميع (Aggregation)**: يقوم الوكيل بقراءة البيانات من SQLite المحلي وضغطها (Gzip).
4. **الرد (Response)**: يتم إرسال البيانات للتطبيق.

---

## 🛡️ حدود الثقة والمعالجة (Trust Boundaries)

- **الراوتر**: هو مصدر الحقيقة للبيانات الخام.
- **التطبيق**: هو المسؤول عن عرض البيانات وإدارة تفضيلات المستخدم.
- **قناة الاتصال**: يتم افتراض أنها "غير آمنة" (LAN مختلطة)، لذا يتم الاعتماد على التوقيع الرقمي (Signature) وليس فقط التشفير.

---

## 📉 استراتيجية التعافي (Recovery Strategy)

- **Safe Fallback**: في حال فشل الوكيل في الرد، يعود التطبيق تلقائياً لاستخدام `LuCI RPC` الأساسي لضمان استمرارية عرض السرعة الحالية على الأقل.
- **Manual Reset**: يمكن للمستخدم إعادة تشغيل الوكيل من داخل إعدادات التطبيق (عبر SSH أو LuCI command).

---

## 🚀 استراتيجية التوسع (Scaling Strategy)

- **Multi-Router Multi-Node**: النظام مصمم ليدعم عدة عملاء (Agents) على عدة راوترات، حيث يتم ربط كل وكيل بـ `Router Identifier` فريد في قاعدة بيانات التطبيق.
- **Plugin Marketplace Ready**: الوكيل يوفر واجهة لإضافة "Scripts" خارجية (Plugins) لجمع بيانات نوعية (مثل درجات حرارة أجهزة معينة).

---
**القرار المعماري**: عزل الوكيل عن خدمات الشبكة الحساسة للراوتر. الوكيل يقرأ البيانات فقط ولا يغير إعدادات التوجيه (Routing) إلا من خلال واجهات معتمدة ومراقبة.
