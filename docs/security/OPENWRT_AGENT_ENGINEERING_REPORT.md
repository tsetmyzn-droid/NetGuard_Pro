## 🧐 التحليل (Analysis)

### 1. Python CGI
**الرأي**: غير عملي للراوترات المنزلية العادية (مثل تلك التي تملك 64MB أو 128MB رام). استهلاك الذاكرة الأساسي لمفسر Python يستهلك جزءاً كبيراً من الموارد المخصصة للشبكة.

### 2. Shell (Ash)
**الرأي**: ممتاز للوظائف البسيطة، ولكن بناء نظام API معقد وآمن (HMAC/Json) باستخدام Shell Script يعتبر كابوساً من حيث الصيانة وعرضة للأخطاء الأمنية.

### 3. Lua + uhttpd (الفائز الحالي)
**الرأي**: Lua هي اللغة الطبيعية لـ OpenWrt و LuCI. هي سريعة بشكل مذهل، استهلاكها للذاكرة ضئيل، وتدعم معالجة JSON وتشفير HMAC عبر مكتبات `nixio` و `libucode` المدمجة.

## 🏆 التوصية النهائية (Recommendations)

### 🥇 الخيار الأفضل حالياً (Current Implementation)
**Lua + uhttpd**
- **السبب**: استغلال البنية التحتية الموجودة مسبقاً في الراوتر (uhttpd). يوفر الاستقرار والموثوقية بدون إضافة وزن (Overhead) للنظام.

### 🚀 الخيار الأفضل مستقبلاً (Long-term Option)
**Rust Microservice**
- **السبب**: عندما ننتقل لمرحلة "Enterprise Gateway" حيث تتوفر موارد أكثر (256MB+ رام ومعالجات ARMv8)، ستوفر Rust أداءً لا يضاهى وأماناً تاماً للذاكرة (Memory Safety) لنظام تتبع مكثف.

---

## 📜 مواصفات التنفيذ المقترح (Lua Implementation)

1. **الواجهة**: ملف Lua CGI يوضع في `/www/cgi-bin/netguard`.
2. **المكتبات**: استخدام `lucid` أو `luci.jsonc` للتعامل مع البيانات.
3. **التواصل مع النواة**: استدعاء أوامر `ubus` و `nlbw` مباشرة عبر `io.popen`.
4. **الأمان**: التحقق من الـ HMAC قبل تنفيذ أي عملية Logic.

---
**الخلاصة**: سنبدأ بـ **Lua** لضمان عمل الـ Agent على أوسع نطاق ممكن من الراوترات (بما فيها القديمة)، مع الحفاظ على هيكلية كود نظيفة تسمح بالانتقال لـ Rust مستقبلاً.
